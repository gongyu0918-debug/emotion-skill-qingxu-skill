# 情绪.skill 思路链审查稿

这份文档的用途很单纯：把项目最初的一串人类思路压缩成一条可以被别人审查的逻辑链。

目标不是证明它已经完美，目标是让别人一眼看见：

- 它在假设什么
- 它想解决什么
- 它把哪些心理学和语言学信号映射成了哪些 Agent 动作
- 它可能在哪些地方过度解读

## 1. 原始目标

让 AI 学会察言观色，读空气。

具体到 Agent 场景，就是让模型从“只理解任务内容”升级到“理解这个状态下的人在发出这个任务”。

## 2. 第一层假设

文本里存在稳定的情绪和姿态线索。

这些线索不一定是显性情绪词，也可能是：

- 标点变化
- 输入长度骤降
- 错别字和拼写错误增加
- 反复强调同一件事
- 轻度修正词
- 延迟带来的语气变化
- 成功后要求收口的表达

## 3. 第二层假设

情绪在这里的价值不是“让模型共情”，而是“让模型切换工作模式”。

也就是：

- 情绪信号不直接决定内容真假
- 情绪信号决定路由、优先级、验证、解释深度、收口动作

## 4. 情绪分类思路

项目保留的核心情绪轴：

- `urgency`
- `frustration`
- `confusion`
- `skepticism`
- `satisfaction`
- `cautiousness`
- `openness`

这几个维度可以并存。

`dominant_mode` 只负责当前轮编排，不代表其他情绪不存在。

## 5. 前置层思路

前置层只做快筛，重点看：

- 词汇
- 标点
- 长短
- 重复
- 延迟
- 卡顿
- 重试
- 同问题反复出现
- 与上文的冲突
- 错拼和输入混乱

前置层的优点：

- 快
- 便宜
- 可解释
- 能直接驱动线程和队列

前置层的缺点：

- 对弱情绪转向不敏感
- 冷启动时用户画像太少

## 6. 后置层思路

后置层做一个很短的隐藏反问：

这句话里有没有情绪词、姿态词、修正词、节奏线索？

它重点抓弱信号：

- `不一定`
- `应该`
- `吧`
- `你确定`
- `先给依据`
- `行吧`
- `Fine.`
- `Same issue again.`

后置层的优点：

- 对微弱姿态变化更敏感
- 更适合冷启动校准
- 更适合做长期学习标签

后置层的缺点：

- 更慢
- 更耗 token
- 更容易过度解读

## 7. 动态权重思路

项目当前的核心判断：

冷启动阶段以后置为主，成熟阶段以前置为主。

切换依据不是拍脑袋，而是一致性率：

- 前置标签和后置拆解是否长期吻合
- 当前轮前后置是否方向一致
- 用户画像是否成熟

这意味着每个用户的前置可信度都可以不一样。

## 8. 工作模式映射思路

情绪层映射到 Agent 的实际动作：

| 情绪主态 | 工作模式 |
|---|---|
| `urgent` | 主线程优先、压后 heartbeat、缩短进度更新时间 |
| `frustrated` | 先修再解释、提高验证等级 |
| `skeptical` | 先给依据和校验点 |
| `cautious` | 验证优先、收紧 scope、降低并发 |
| `confused` | 更耐心解释、分步骤、显式假设 |
| `satisfied` | 进入 guard mode |
| `exploratory` | 给多方案和 tradeoff |

## 9. 为什么 guard mode 很关键

很多 Agent 场景里，最危险的时候不是“还没解决”，而是“已经解决以后还在继续改”。

所以满意态不应该只被理解成“用户开心了”，还应该被理解成：

- 当前结果大概率可接受
- 接下来的重点从推进改成收口
- 需要防配置漂移、防回归、防额外破坏

## 10. 项目最核心的价值主张

情绪层不是语气美化器。

它的核心价值是这四件事：

1. 改任务优先级
2. 改验证强度
3. 改解释和响应方式
4. 改成功后的收口策略

## 11. 目前最容易出问题的地方

这套链条当前最值得被挑战的点：

1. 弱质疑和困惑的边界
   - `不一定` 有时是质疑，有时只是继续讨论
2. 礼貌和高压同时存在时的权重
   - `麻烦尽快处理一下，谢谢`
3. 中文冷淡语气的复杂性
   - `行吧`
   - `算了`
   - `呵`
4. 后置反问过度解释的风险
5. 不同文化圈里标点和句号的语气差异

## 12. 建议审查问题

如果你要检查这套逻辑，最值得看这几个问题：

1. 哪些信号是情绪，哪些信号其实只是写作风格？
2. 哪些路由动作太激进，可能会误伤正常对话？
3. 哪些情绪其实应该拆成姿态，而不是情绪？
4. 哪些维度应该是长期人格，哪些应该是短期状态？
5. 满意态进入 guard mode 的触发条件会不会太早？

## 13. 当前立场

这个项目当前属于研究型公开实现。

它已经足够展示价值：

- 前置快筛
- 后置校准
- 一致性驱动权重
- 线程与验证联动
- 用户级长期学习

它也还保留几个明确缺口：

- live A/B 还不够
- 弱质疑主导态仍需校准
- 多文化语气样本还可以继续扩

这份文档本身就是给别人挑错用的。
